### Design decisions

- **Three-zone lifecycle.** Split states into *gate* states (`prepared`, `pending_approval`,
  `approved`, `rejected`, `expired`) that a future implementation of this design could reach in
  the approval flow, and *launch* states (`running`, `completed`, `failed`, `cancelled`) that are
  parked and execute nothing in V0.1c but are specified for forward schema stability.
  `interrupted` folded into `cancelled` via a `cancel_reason` discriminator.
- **Bind approval to an exact intent.** Introduced `run_intent_hash` over
  `(role_hash, policy_hash, argv_hash, prompt_hash, cwd, timeout_seconds, max_turns,
  max_budget_usd)` so any drift invalidates an approval; the launch path re-computes and
  re-checks rather than trusting the stored artifact.
- **Hashes, not raw values.** The approval artifact stores `prompt_hash` + `prompt_len` and the
  redaction policy string, never raw prompt/env/stderr/secrets. File mode `0600`, run dir `0700`,
  tamper-as-refuse.
- **Single-use approval.** `one_time_nonce` + `consumed_at` make approval one-shot; replay and
  double-consume fail closed. Both a request TTL (`expires_at`) and an approval TTL are enforced.
- **Default-refuse launch.** The only launch edge (`approved -> running`) is hard-disabled: with
  no separately-approved future-phase launch flag, fail-closed rule 8 is always true, so
  `run-approved` always refuses in V0.1c.
- **Honest claims.** The gate is a process/policy control, not an OS sandbox; humanness is an
  attestation + out-of-band control, not cryptographic proof; `business_verdict` stays `null` and
  `completed` is never a business PASS. These mirror existing repo invariants.
