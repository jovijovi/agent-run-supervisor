# agent-run-supervisor V0.1c HITL / Manual Real-Run Design Gate Plan

> **For Hermes:** Claude Code is the main design/architecture worker for this docs/design
> gate. Codex CLI is the primary independent reviewer. Hermes controls scope, verification,
> evidence, and arbitration. **This is a design/docs-only gate — no implementation, no real
> launch, and no source behavior change.**

**Goal:** Produce the design contract for a future, separately-approved one-shot real AGENT
run: a human-in-the-loop (HITL) manual approval gate with a full lifecycle, a binding redacted
approval artifact, fail-closed launch rules, a proposed (not implemented) CLI/API surface, a
test plan, and a boundary review checklist. Keep the real-launch edge hard-disabled.

**Architecture:** Pure documentation/design. No source or runtime change. The design extends
the V0.1b posture (version-only `doctor` probes, `allowed_roots` as cwd/config gate, stable
real-run refusal) by specifying *what an approval would have to look like* before any future
phase could consider enabling launch.

**Tech Stack:** Markdown docs under `docs/`; stdlib doc tools `tools/build_docs_index.py` and
`tools/docs_drift_signal.py`; Python 3.11 for the doc gates only.

## Context

The dashboard's "next allowed request" in `docs/roadmap/current-status.md` is *"plan a
HITL/manual real-run design gate"*, design-only, single-agent, manual, budget/timeout/max-turn
bounded, no auto-routing by default. The human approved starting V0.1c **strictly as a design
gate**. Phase -1, V0.1a (exec-only), and V0.1b (real-run preflight hardening) are complete on
`main`. No real AGENT launch exists; `run` without `--no-real-run` refuses; `allowed_roots` is
not a sandbox; `business_verdict` is always `null`.

## Explicit non-approvals

This phase does **not** approve, implement, or imply:

- real AGENT execution or any acpx `exec`/launch for `run`;
- persistent sessions, session registry, locking, stale-lock recovery, or multi-turn retention;
- Sachima integration; real AGENT automatic replies;
- public ingress or real IM delivery;
- Gateway restart/reload/replace or any Gateway lifecycle operation;
- production config writes; live/default-on behavior;
- worker auto-routing; agent-to-agent automatic routing;
- participant persistence or management UI; `@all` fanout;
- trusted Markdown/HTML rendering;
- treating `allowed_roots` as an OS/filesystem sandbox; any OS sandbox/isolation proof.

## Risks and open questions

- **Risk:** a design doc reads as an implementation approval. **Control:** every CLI/API and
  launch state is marked proposed/parked; non-approvals repeated; GOAL/current-status use
  design-only wording.
- **Risk:** the approval gate is mistaken for a security sandbox. **Control:** the design states
  the gate is a process/policy control, keeps the `allowed_roots` disclaimer, and parks
  containment under `ARS-SANDBOX-BOUNDARY`.
- **Risk:** humanness is overclaimed. **Control:** the design states human origin is an
  attestation + out-of-band control, not cryptographic proof.
- **Open question:** exact `request_ttl`/`approval_ttl` defaults, nonce storage location, and
  budget-accounting mechanics are deferred to the future implementing phase; this gate only
  fixes the contract and the fail-closed rules.
- **Open question:** whether `run-approved` should be a separate binary/subcommand or remain
  fused with `run` behind the future-phase flag — left to the implementing phase.
