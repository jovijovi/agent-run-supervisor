# agent-run-supervisor V0.1a Revised Design Gate

## Goal

Define the first safe implementation boundary for `agent-run-supervisor`: an independent Python library plus dev CLI that supervises external AGENT runs through pinned `acpx@0.10.0`, normalizes ACP/JSON streams, and records auditable run artifacts.

This revision incorporates the Codex CLI review, prior Claude Code review themes around lifecycle/boundaries/testability, and the verified `acpx@0.10.0` exit-code model.

## Non-goals / explicit non-approvals

V0.1a does **not** implement or imply:

- persistent sessions;
- public ingress;
- real IM delivery;
- Gateway restart/reload/replace;
- production config writes;
- live/default-on behavior;
- worker auto-routing;
- participant persistence or management UI;
- `@all` fanout;
- agent-to-agent automatic routing;
- trusted Markdown/HTML rendering;
- treating `allowed_roots` as a security boundary unless separately proven.

## Positioning

```text
Sachima / Hermes consumer
  -> selects role_id, builds task prompt/context, renders progress, interprets business verdict
agent-run-supervisor
  -> validates AgentRoleSpec, compiles acpx args/policy, runs acpx, classifies exits, parses streams, writes result/audit artifacts
acpx@0.10.0
  -> ACP runner/client and adapter boundary
external AGENT
  -> Codex/Claude/native ACP worker or reviewer
```

`agent-run-supervisor` is not a daemon, not a Hermes profile, and not a Sachima adapter. It owns runner supervision and auditability only.

## Phase split

### Phase -1: acpx contract spike

Purpose: prove the real `acpx@0.10.0` contract before implementing the library.

Required evidence:

- `acpx --version` and exact binary/adapter provenance.
- Command grammar for exec mode using the exact future runner flag set: `--format json --json-strict --suppress-reads --timeout <timeout_seconds> --max-turns <max_turns>` plus the compiled permission flags/policy.
- Real policy JSON accepted by `--permission-policy`.
- Permission kind names and default behavior.
- Observed acpx JSON stdout success fixture captured from the exact runner flag combination above. Do not capture a different "raw ACP" stream and then write the V0.1a parser against a stream the runner will never emit.
- A written `AcpxStdoutSchema` note stating whether `acpx@0.10.0` JSON mode emitted raw JSON-RPC lines such as `session/update` / `agent_message_chunk`, or acpx-normalized envelopes such as `type=text|thought|tool_call|done|error`. The parser must use this observed schema, not pre-spike terminology.
- Failure fixtures captured with the same runner flag family where reachable:
  - exit `1` runtime/runner error;
  - exit `2` usage error;
  - exit `3` acpx timeout via `--timeout`;
  - exit `5` permission denied / permission prompt unavailable;
  - exit `130` interrupted, if practical;
  - malformed observed stdout stream;
  - unknown observed update/event variant.
- Management/session-only fixtures kept separate from exec stdout fixtures:
  - exit `4` no session, if reachable only through prompt/session management paths;
  - exit `0` management `status=no-session` fixture.
- Explicit finding on path-level enforcement: either acpx can enforce roots, or V0.1a documents `allowed_roots` as cwd/config validation only.

Exit criteria:

- Fixtures are checked into the future repo under a non-ignored path.
- Replay expectations are written before parser implementation.
- Any unknown or unsupported acpx behavior is captured as a design constraint, not guessed around.

### V0.1a: exec-only vertical slice

Scope:

- `AgentRoleSpec` schema and validation.
- Role-bound acpx args/policy compiler.
- Exec-only subprocess runner.
- `AcpxExitClassifier`.
- Observed acpx JSON stdout parser and normalized events. The parser targets the stdout schema captured by Phase -1 with the real runner flags; it must not assume raw ACP JSON-RPC or normalized envelopes before fixtures prove the shape.
- EventStore with redaction and secure file permissions.
- CLI: `validate-role`, `run`, `replay`, `doctor`.
- Fixture-driven tests and minimal real smoke.

Explicitly deferred:

- persistent sessions;
- session registry;
- sessions CLI;
- locking/stale-lock recovery;
- multi-turn context retention;
- Sachima behavior-bearing integration.

### V0.1b+: persistent/session phase

Only after explicit approval and after Phase -1 proves session command shapes. Must include session isolation, role/workspace hash, lock strategy, stale-lock handling, crash recovery, close failure semantics, and leakage tests.

## AgentRoleSpec V0.1a

Minimal shape:

```yaml
schema_version: 1
role_id: codex-reviewer
display_name: Codex Review 专用
description: 只读代码评审 AGENT

runner:
  type: acpx
  acpx_version: "0.10.0"
  acpx_binary: /abs/path/or/pinned/runner
  adapter_agent: codex
  model: null

workspace:
  default_cwd: /abs/path/to/repo
  allowed_roots:
    - /abs/path/to/repo
  allowed_roots_security_boundary: false

permissions:
  read: true
  search: true
  write: false
  execute: false
  terminal: false
  delete: false
  move: false
  fetch: false
  switch_mode: false
  other: false

session:
  strategy: exec

limits:
  timeout_seconds: 900
  max_turns: 1
  max_output_bytes: 10485760

prompt:
  role_instruction: |
    You are a read-only reviewer.
  output_contract: |
    Return PASS/BLOCK/NOTES.

redaction:
  suppress_reads: true
  redact_prompt: true
  redact_stderr: true
  redact_metadata: true
  redact_env: true
```

Rules:

- `role_id` is stable identity and policy boundary.
- Role permissions are maximum acpx-controllable permissions.
- `allowed_roots` validates caller cwd and labeling only in V0.1a unless a real sandbox/path policy is proven.
- No business-only permissions live here: no delivery, no Gateway lifecycle, no production config, no `@all`.

## Permission model

Compile role permissions into generated acpx policy:

- allowed kinds -> `autoApprove`;
- denied kinds -> `autoDeny`;
- `defaultAction=deny`;
- `terminal=false` -> `--no-terminal`;
- always `--non-interactive-permissions fail`;
- default automation flags: `--format json --json-strict --suppress-reads --timeout <limits.timeout_seconds> --max-turns <limits.max_turns>`.
- supervisor uses an outer watchdog longer than acpx `--timeout` to catch hung acpx/adapters without stealing acpx's own exit `3` path.

Security claim:

- The compiler can constrain acpx-declared tool classes.
- It must not claim OS/filesystem isolation unless Phase -1 proves path-level acpx enforcement or an external sandbox is added.

## Runner and status model

`AcpxRunner` owns subprocess execution and records:

- argv;
- cwd;
- acpx version;
- role hash;
- policy hash;
- start/end timestamps;
- exit code;
- signal;
- stdout/stderr paths;
- timeout/kill reason.

`AgentRunStatus` is supervisor-owned:

```text
completed
runner_error
invalid_invocation
timed_out
no_session
permission_denied
interrupted
protocol_error
infrastructure_error
policy_error
```

`completed` means runner/protocol completed. It does not mean the AGENT's business verdict is PASS. V0.1a does not parse business PASS/BLOCK at all: `business_verdict` is always `null`, and consumers that own a business contract may interpret `final_message` outside supervisor.

## acpx@0.10.0 exit-code mapping

Classification table:

| exit | acpx meaning | supervisor status |
|---:|---|---|
| 0 | command exited successfully | `completed` only after protocol/result validation |
| 1 | generic runtime/runner error | default `runner_error`; classify as `infrastructure_error` only for supervisor/OS failures such as spawn failure, watchdog kill, disk/permission failure, or explicit infrastructure `acpxCode`/origin evidence |
| 2 | usage error | `invalid_invocation` |
| 3 | acpx timeout from `--timeout` | `timed_out` |
| 4 | no session | `no_session` (V0.1a exec path should not normally produce this; primarily management/persistent-session fixture coverage) |
| 5 | permission denied or permission prompt unavailable | `permission_denied` |
| 130 | interrupted | `interrupted` |
| other nonzero | unknown failure | `infrastructure_error` |

Additional rules:

- JSON-RPC/acpx error `data.acpxCode` overrides or refines bare exit-code classification.
- Nonzero exit never becomes `completed` or user-visible PASS.
- Exit `0` still requires observed stdout/schema validation.
- Supervisor-enforced timeout/kill is recorded separately from acpx exit `3` if acpx never returned.
- `origin=queue` and exit `4 no_session` are V0.1b/session-management concepts in normal use; V0.1a keeps them only as reserved classifier/fixture cases so future session work does not change result schema.
- `acpx status/sessions/...` management JSON is a separate schema. Example: `status=no-session` can exit `0` and is not an exec success.

Retryability matrix for V0.1a:

| status | default retryable | notes |
|---|---:|---|
| `completed` | false | no retry needed; business interpretation is outside V0.1a |
| `invalid_invocation` | false | caller/spec bug |
| `permission_denied` | false | includes prompt unavailable; retry only after role/policy/user approval changes |
| `policy_error` | false | role/policy compilation bug |
| `no_session` | false in V0.1a | reserved/session-management issue, not an exec retry path |
| `interrupted` | false | user/operator cancelled |
| `timed_out` | true | retry only with caller backoff/budget and same non-approval boundaries |
| `runner_error` | true when origin is `runtime`/`acp` and no policy/usage evidence; false when deterministic adapter/config failure is proven |
| `protocol_error` | false by default | retry only after parser/acpx drift investigation |
| `infrastructure_error` | true for transient OS/process/disk/network symptoms, false for deterministic missing binary/version mismatch |

## Normalized events

V0.1a event types:

```text
run_started
agent_message_delta
tool_started
tool_updated
tool_completed
usage_updated
permission_requested
permission_denied
unknown_update
run_completed
run_failed
```

Parser requirements:

- parse only the observed acpx JSON stdout schema captured by Phase -1 with the real runner flags;
- tolerate optional fields inside that observed schema;
- fail closed on malformed observed stdout framing;
- preserve unknown update/event type plus key path and value type summary only; do not persist raw unknown values unless debug raw capture is explicitly enabled and marked unsafe;
- assemble `final_message` from the observed text-delta source: if Phase -1 proves raw JSON-RPC, use ordered `session/update` text chunks; if it proves acpx envelopes, use ordered `type=text` payloads. Do not support a schema that was not fixture-proven.
- separate runner/protocol completion from business PASS/BLOCK parsing; in V0.1a there is no business parsing.
- enforce `max_output_bytes`; on overflow stop parsing, mark `truncated=true` and `truncate_reason=max_output_bytes`, and return `protocol_error` unless Phase -1 defines a safe completed-with-truncation fixture.

## EventStore and redaction

Run layout:

```text
.agent-run-supervisor/runs/<run_id>/
  metadata.json
  prompt.txt
  env.redacted.json
  command.argv.json
  generated-policy.json
  acpx-stdout.ndjson
  normalized-events.jsonl
  stderr.log
  result.json
  redaction-report.json
```

Requirements:

- run directory mode `0700`;
- final artifact mode `0600`;
- atomic write/rename for final JSON artifacts;
- append-only discipline for stream files;
- redaction covers prompt, stderr, metadata, argv, env, observed acpx stdout stream, normalized events, final result;
- raw preservation requires explicit debug opt-in and must be marked unsafe for chat/display;
- retention/cleanup knobs before long-lived use.

Lifecycle and timeout handling:

- pass `limits.timeout_seconds` to acpx as `--timeout` so acpx can produce its own exit `3` path;
- set supervisor outer watchdog to `timeout_seconds + grace_seconds` or a documented larger bound;
- on outer watchdog expiry, send SIGTERM to the acpx process group when supported, wait a fixed grace period, then send SIGKILL;
- record `kill_reason`, `kill_signal`, `grace_ms`, `process_group_used`, and whether stdout/stderr were truncated or closed cleanly in metadata/result;
- classify supervisor kill as `infrastructure_error` or `timed_out` with `origin=supervisor`, not as acpx exit `3`.

## CLI

V0.1a commands:

```bash
agent-run-supervisor validate-role <role-file>
agent-run-supervisor run --role <role-file-or-id> --prompt-file <file> [--cwd <dir>]
agent-run-supervisor replay <events.ndjson>
agent-run-supervisor doctor
```

`doctor` checks:

- acpx binary and exact version;
- Node version meets the observed acpx requirement (`>=22.12` unless Phase -1 proves a different exact minimum);
- adapter availability;
- whether runtime `npx` fetch would occur;
- policy dry-run/parseability;
- fixture replay;
- cwd/allowed_roots validation;
- EventStore permissions;
- redaction probe.

## Testing strategy

Test layers:

1. Schema tests for `AgentRoleSpec` valid/invalid roles.
2. Policy compiler golden tests.
3. Exit classifier table tests for `0/1/2/3/4/5/130/unknown`.
4. Parser fixture replay tests for all Phase -1 observed stdout fixtures, captured with the exact V0.1a runner flags including `--timeout` and `--max-turns`.
5. EventStore permission/atomic-write tests.
6. Redaction tests for prompt/stderr/env/argv/metadata/event/result leakage.
7. Runner tests with fake subprocesses, including outer-watchdog SIGTERM/SIGKILL behavior.
8. One real `acpx@0.10.0` smoke in scratch repo, no tools, no edits.

Acceptance gate:

- all fixtures replay deterministically;
- fixture capture used the same runner flags the V0.1a runner will use, especially `--format json --json-strict --suppress-reads --timeout --max-turns`;
- the parser document names the observed acpx stdout schema and does not mix raw ACP terms with envelope output unless both are separately fixture-proven;
- no nonzero exit can produce `completed`;
- malformed observed stdout yields `protocol_error`;
- status-management JSON cannot be parsed as exec success;
- permission denied yields `permission_denied` even if partial text exists;
- `business_verdict` remains `null` in V0.1a;
- unknown updates preserve key/type summary only, not raw values;
- max-output overflow sets `truncated=true` and a deterministic failure/status path;
- docs state the allowed_roots boundary honestly.

## Sachima integration boundary

Later Sachima integration should remain thin:

```text
mention -> role_id
build context pack / prompt
call agent-run-supervisor run
render NormalizedEvent progress
interpret AgentRunResult.final_message and business verdict
```

Sachima must not parse ACP/acpx directly, own runner lifecycle, or infer external delivery approval from a supervisor result.

## Design recommendation

Proceed only with:

```text
Phase -1 acpx contract spike -> V0.1a exec-only vertical slice
```

Do not start persistent sessions or behavior-bearing Sachima integration in the same batch. That would mix three risk surfaces: acpx contract drift, lifecycle/session leakage, and user-facing agent behavior. Bad cocktail, cute umbrella notwithstanding.
