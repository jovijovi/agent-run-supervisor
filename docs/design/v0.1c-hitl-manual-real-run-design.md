# agent-run-supervisor V0.1c HITL / Manual Real-Run Design Gate

> **Deprecated product direction.** This document is retained as historical design context only.
> It introduced per-run manual approval semantics, but the current product decision is
> role-bound authorization through `AgentRoleSpec`, governed by `docs/design/v0.1a-design.md`,
> `docs/product/prd.md`, and `docs/design/technical-solution.md`.
> Do not use this document as implementation authority unless the user explicitly reverses
> the role-bound authorization decision.

> **Design-only.** This document specifies the *contract* for a future, separately
> approved one-shot real AGENT run. It does **not** approve, implement, or imply real
> launch. Every CLI/API and runtime state below is marked **PROPOSED** or **FUTURE**.
> Nothing here grants permission to spawn an agent, open a session, or deliver output.

## Goal

Define a human-in-the-loop (HITL) **manual approval gate** that a future phase could use
to authorize exactly one (`max_turns`-bounded, budget-bounded, timeout-bounded) real
AGENT run. The gate must:

- separate **intent preparation** (safe, no launch) from **human approval** (no launch)
  from **launch** (parked, future-phase only);
- bind an approval to an *exact* run intent so that any drift in role, policy, argv,
  prompt, or cwd invalidates the approval;
- be **fail-closed**: the default answer to "may I launch?" is always *no*, and every
  missing/contradictory precondition keeps it *no*;
- produce **redacted, restrictively-permissioned, auditable** approval artifacts that
  never contain raw prompt text, environment values, secrets, or raw stderr;
- make no claim of OS/filesystem sandboxing and no claim that runner completion equals a
  business PASS.

This builds directly on V0.1b: `doctor` already probes Node/acpx without launching,
`allowed_roots` is already a cwd/config gate (not a sandbox), and `run` without
`--no-real-run` already returns a stable refusal. V0.1c specifies *what an approval would
have to look like* before any future phase is even allowed to consider flipping launch on.

## Non-goals / explicit non-approvals

V0.1c is a design gate. It does **not** approve, implement, or imply:

- real AGENT execution or any acpx `exec`/launch for `run`;
- persistent sessions, session registry, locking, stale-lock recovery, or multi-turn
  context retention;
- Sachima behavior integration;
- real AGENT automatic replies;
- public ingress or real IM delivery;
- Gateway restart/reload/replace or any Gateway lifecycle operation;
- production config writes;
- live / default-on behavior;
- worker auto-routing or agent-to-agent automatic routing;
- participant persistence, participant UI, or `@all` fanout;
- trusted Markdown/HTML rendering;
- treating `allowed_roots` as an OS/filesystem sandbox;
- any OS-level sandbox or isolation proof.

The approval gate is a **process and policy control**, not an OS enforcement mechanism.
`allowed_roots` is not an OS/filesystem sandbox; it remains cwd/config-intent validation only.
A human approval reduces the chance of an unintended launch; it does not contain a launched
agent. Containment remains parked under a separate sandbox phase (`ARS-SANDBOX-BOUNDARY`).

## Positioning

```text
Human operator (approver)
  -> reviews a prepared run intent, approves or rejects out-of-band, owns the decision
agent-run-supervisor (this project, FUTURE launch parked)
  -> prepares run intent, computes binding hashes, writes redacted approval artifact,
     re-validates everything at launch time, fails closed unless every precondition holds
acpx@0.10.0 / ACP runner boundary (NOT invoked in V0.1c)
  -> would launch the external AGENT only in a future approved phase
external AGENT
  -> Codex / Claude / native ACP worker or reviewer
```

The gate sits entirely on the supervisor side. The approver acts out-of-band (CLI invoked
by a human, or an external system the human controls). The supervisor never infers approval
from its own state, from a consumer, or from an agent.

## Lifecycle / state machine

The approval artifact is a small state object identified by `run_id`. States below are split
into **gate states** (the only states a future implementation of *this design* would reach
in the approval flow) and **launch states** (specified for forward schema stability but
**parked** under `ARS-REAL-RUN-GATE`; they execute nothing in V0.1c).

### States

| State | Class | Meaning | Launch possible? |
|---|---|---|---|
| `prepared` | gate | Run intent compiled and validated; binding hashes computed; artifact written. No approval yet. | No |
| `pending_approval` | gate | Prepared intent submitted for a human decision; `expires_at` is set; awaiting approve/reject. | No |
| `approved` | gate | A human approved this exact intent; `approval_id`/`one_time_nonce` issued; not yet consumed. | Only via future-phase launch flag (parked) |
| `rejected` | gate (terminal) | A human rejected; `reason` recorded. | No |
| `expired` | gate (terminal) | `expires_at` (or approval TTL) passed before consumption. Must re-prepare. | No |
| `running` | launch (FUTURE/parked) | Nonce consumed (`consumed_at` set), one-shot launch in progress. | n/a — parked |
| `completed` | launch (FUTURE/parked) | Runner/protocol completed. **Not** a business PASS; `business_verdict` stays `null`. | n/a — parked |
| `failed` | launch (FUTURE/parked) | Runner error / timeout / protocol / infrastructure failure. | n/a — parked |
| `cancelled` | launch (FUTURE/parked) | Operator cancelled, or process interrupted (signal/`130`). | n/a — parked |

`interrupted` is folded into `cancelled` with a `cancel_reason` discriminator
(`operator_cancel` vs `signal_interrupt`) to keep the schema small.

### Transitions

```text
prepare-run        : (none)            -> prepared
submit (implicit)  : prepared          -> pending_approval
approve-run        : pending_approval  -> approved        [human only, sets nonce + TTL]
reject-run         : pending_approval  -> rejected        [terminal]
ttl/clock expiry   : pending_approval  -> expired         [terminal]
                     approved (unused) -> expired          [terminal, approval TTL]
--- launch states below are PARKED (future phase only) ---
consume (launch)   : approved          -> running         [requires future-phase flag;
                                                            burns nonce; re-validates all hashes]
runner result      : running           -> completed | failed | cancelled  [terminal]
```

Rules:

- The **only** edge that can ever start a real agent is `approved -> running`, and in V0.1c
  that edge is hard-disabled (no future-phase launch flag exists; the launch path refuses).
- All terminal states are immutable records. Re-running requires a fresh `prepare-run`
  (a new `run_id`, new hashes, new nonce).
- A `prepared` or `pending_approval` artifact never authorizes anything; it is a request.

## Approval artifact shape (PROPOSED schema)

One artifact per `run_id`. Stored as redacted JSON, file mode `0600`, in a run directory
with mode `0700` (consistent with V0.1a EventStore). The artifact stores **hashes, not raw
values** for all sensitive inputs.

```json
{
  "schema_version": 1,
  "run_id": "run_...",
  "state": "pending_approval",

  "role_id": "codex-reviewer",
  "role_hash": "sha256:<hex>",
  "cwd": "/abs/effective/cwd",
  "allowed_roots_disclaimer": "allowed_roots validates cwd/config intent only; it is NOT an OS/filesystem sandbox",
  "allowed_roots_security_boundary": false,

  "policy_hash": "sha256:<hex>",
  "run_intent_hash": "sha256:<hex>",
  "argv_hash": "sha256:<hex>",
  "prompt_hash": "sha256:<hex>",
  "prompt_redaction_policy": "raw prompt never stored; hash over canonicalized UTF-8 bytes; prompt_len recorded for sanity only",
  "prompt_len": 1234,

  "timeout_seconds": 900,
  "max_turns": 1,
  "max_budget_usd": 0.50,

  "created_at": "2026-05-29T08:20:00+0800",
  "expires_at": "2026-05-29T08:35:00+0800",

  "approved_by": null,
  "approved_at": null,
  "approval_id": null,
  "one_time_nonce": null,
  "consumed_at": null,

  "reason": null,

  "supervisor_version": "0.1c-design",
  "launched_real_agent": false
}
```

Field notes:

- **`run_id`** — stable identity for the request; also the run-directory name.
- **`role_id` / `role_hash`** — `role_hash` is SHA-256 over the canonicalized
  `AgentRoleSpec` (same role-hash concept used in V0.1a runner metadata). `role_id` is
  human-readable; `role_hash` is the binding value.
- **`cwd`** — the resolved effective cwd that *would* be used. Re-validated at launch, never
  trusted from the artifact alone.
- **`allowed_roots_disclaimer` / `allowed_roots_security_boundary`** — carried verbatim so no
  reader can mistake the gate for a sandbox.
- **`policy_hash`** — SHA-256 over the compiled acpx permission policy JSON.
- **`run_intent_hash`** — SHA-256 over the canonical tuple
  `(role_hash, policy_hash, argv_hash, prompt_hash, cwd, timeout_seconds, max_turns, max_budget_usd)`.
  This is the single value an approval is bound to; it is the cheapest thing to re-check.
- **`argv_hash`** — SHA-256 over the compiled acpx argv list (the exact future runner flags).
- **`prompt_hash` / `prompt_redaction_policy` / `prompt_len`** — the raw prompt is **never**
  stored in the artifact. Only its hash, the policy string, and a length integer.
- **`timeout_seconds` / `max_turns` / `max_budget_usd`** — hard caps. All three are required
  and must be positive; absence or non-positive values fail closed.
- **`created_at` / `expires_at`** — request window. `expires_at = created_at + request_ttl`.
- **`approved_by`** — human approver identifier/attestation (e.g. operator id). Non-human or
  empty values fail closed at launch.
- **`approved_at`** — set on approval; an approval TTL (`approved_at + approval_ttl`) is also
  enforced so a stale approval cannot be consumed much later.
- **`approval_id` / `one_time_nonce`** — issued on approval. The nonce is single-use; launch
  consumes it exactly once.
- **`consumed_at`** — set when (in a future phase) the nonce is burned. Non-null means the
  approval is spent; re-use fails closed.
- **`reason`** — free-text rejection/approval note; redaction-scanned like any other field.
- **`launched_real_agent`** — always `false` in V0.1c; present so audit tooling has a stable
  field name when a future phase exists.

## Fail-closed rules

The (future, parked) launch path computes the answer to "may I launch `run_id`?" as a
logical AND of every check below. **Any** failing check returns a stable refusal and starts
nothing. The default — with no future-phase launch flag — is an unconditional refusal
regardless of artifact state.

1. **No approval** — artifact missing, unreadable, or `state != approved` → refuse.
2. **Expired approval** — `now > expires_at`, or `now > approved_at + approval_ttl` → refuse
   and transition to `expired`.
3. **Hash mismatch** — recompute `role_hash`, `policy_hash`, `argv_hash`, `prompt_hash`, and
   `run_intent_hash` from current inputs; if any differs from the approved artifact → refuse.
4. **cwd outside allowed_roots** — re-validate the effective cwd against the role's
   `allowed_roots` at launch time (do not trust the stored `cwd`); outside → refuse.
5. **Missing/invalid caps** — `timeout_seconds`, `max_turns`, or `max_budget_usd` absent,
   non-numeric, or non-positive → refuse.
6. **Role mismatch** — `role_id`/`role_hash` at launch differ from the approved values, or the
   role no longer validates → refuse.
7. **Nonce replay / already consumed** — `one_time_nonce` missing, unknown, or already burned
   (`consumed_at` set, or nonce not in the unconsumed set) → refuse.
8. **No future-phase launch flag** — the explicit, separately-approved launch flag for a real
   run is not enabled → refuse. **This is the current default and is always true in V0.1c.**
9. **Non-human / unattested origin** — `approved_by` empty, or the approval origin is the
   supervisor itself, a consumer, or an agent → refuse. (This is an attestation + out-of-band
   control, not cryptographic proof of humanness; the design states this honestly.)
10. **Tamper / permission signals** — artifact world-readable/writable, wrong file mode,
    malformed JSON, or `schema_version` mismatch → refuse.

Refusal is observable and deterministic (stable code + message) so tests and a future HITL UI
can rely on it. Refusal on the launch path must create no run artifacts and launch no process.

## CLI / API surface (PROPOSED, NOT IMPLEMENTED)

All commands below are **proposals** for a future phase. None are implemented in V0.1c. The
only launch-capable command (`run-approved`) is specified as **always refusing** until a
separate phase is approved.

```bash
# Safe: compiles + validates intent, computes hashes, writes a `prepared` artifact. No launch.
agent-run-supervisor prepare-run \
  --role <role-file-or-id> --prompt-file <file> [--cwd <dir>] \
  --timeout <seconds> --max-turns <n> --max-budget-usd <amount>

# Human-only: pending_approval -> approved. Issues approval_id + one_time_nonce + TTL. No launch.
agent-run-supervisor approve-run <run_id> --approved-by <operator-id> [--ttl <seconds>] [--reason <text>]

# Human-only: pending_approval -> rejected (terminal). No launch.
agent-run-supervisor reject-run <run_id> --reason <text>

# Read-only auditing of a redacted artifact / artifacts. No launch.
agent-run-supervisor show-run <run_id>
agent-run-supervisor list-runs

# PARKED: would consume the nonce and start one real run in a FUTURE approved phase.
# In V0.1c this MUST fail closed (no future-phase launch flag) and start nothing.
agent-run-supervisor run-approved <run_id>
```

Surface rules:

- `prepare-run`, `approve-run`, `reject-run`, `show-run`, `list-runs` never launch an agent.
- `approve-run`/`reject-run` are the human decision points; they are the only writers of
  `approved_by`, `approval_id`, `one_time_nonce`, and `reason`.
- `run-approved` is the single gated launch path. Its V0.1c-described behavior is to run every
  fail-closed check, find rule 8 (no future-phase flag) always true, and refuse.
- Help text for `run-approved` must state plainly that it refuses real launch in the current
  phase (mirroring the V0.1b `run` help-boundary fix), with no wording implying optional launch.

## Test plan (for the future implementing phase)

Layered, fixture-driven, stdlib + pytest, consistent with V0.1a/V0.1b. Each fail-closed rule
gets at least one negative test.

1. **Artifact golden tests** — `prepare-run` produces a deterministic redacted artifact whose
   field set matches the schema; hashes are stable for stable inputs and change when any bound
   input changes (role, policy, argv, prompt, cwd, caps).
2. **State-machine tests** — every legal transition succeeds; every illegal transition (e.g.
   approve a `rejected`/`expired` artifact, reject an `approved` one) fails closed.
3. **Fail-closed path tests** — one test per rule in *Fail-closed rules* (1–10): no approval,
   expired, each hash mismatch, cwd outside roots, each missing/non-positive cap, role
   mismatch, nonce replay/double-consume, missing future-phase flag, non-human/empty
   `approved_by`, tamper/permission/malformed-JSON/schema-mismatch.
4. **Redaction tests** — no raw prompt, no env values, no secrets, no raw stderr in any
   artifact, log, or refusal message; only hashes + `prompt_len`; secret-shaped scan over
   generated artifacts returns zero.
5. **File-permission tests** — run dir `0700`, artifact `0600`; a world-readable artifact is
   treated as tampered and fails closed.
6. **No-artifact-on-refusal tests** — `run-approved` refusal (and any pre-approval validation
   failure that should not persist) creates no run directory and no partial artifact, mirroring
   the V0.1b real-run refusal assertions.
7. **No-launch tests** — assert `launched_real_agent=false` everywhere; assert no acpx `exec`
   / subprocess launch is reachable from any command in this surface; version-only probes (from
   V0.1b `doctor`) remain the only allowed external calls.
8. **No-sandbox-claim tests** — artifact, help text, and docs keep
   `allowed_roots_security_boundary=false` and the disclaimer string; assert no output claims OS
   isolation.
9. **Honesty tests** — `business_verdict` stays `null`; `completed` is never rendered as a
   business PASS; refusal messages do not over-claim safety.

Acceptance for that future phase would additionally require RED/GREEN evidence per check and
independent review (below). None of this is in scope for V0.1c beyond *specifying* it.

## Review checklist (Codex primary / Claude auxiliary / Hermes arbiter)

Boundary review for this design gate. A reviewer should be able to answer every item from the
docs alone.

- [ ] **Scope** — does any doc implement, enable, or imply real launch, sessions, Sachima,
      ingress, delivery, Gateway ops, production config writes, or live/default-on behavior?
      (Must be **no**.)
- [ ] **Fail-closed completeness** — is the default answer always *refuse*? Is every listed
      precondition an AND-gate? Are there reachable states where launch could occur without all
      ten checks? (Must be none.)
- [ ] **Binding integrity** — does an approval bind to exact hashes (role/policy/argv/prompt/cwd/
      caps) such that any drift invalidates it? Is `run_intent_hash` re-checked at launch?
- [ ] **Single-use** — is the nonce provably one-shot (consume-once, replay refused)? Is approval
      TTL enforced in addition to request expiry?
- [ ] **Human origin** — is approval human/out-of-band only, and is the humanness claim stated
      *honestly* as attestation rather than proof?
- [ ] **Redaction** — are raw prompt/env/stderr/secrets excluded from artifacts and messages
      (hashes + length only)? Is the redaction policy explicit?
- [ ] **Permissions** — `0700`/`0600`, atomic write, tamper-as-refuse — specified?
- [ ] **No sandbox overclaim** — is `allowed_roots` consistently described as cwd/config gate,
      not OS isolation? Is containment explicitly parked?
- [ ] **Honesty of status** — `business_verdict=null`, `completed != PASS` preserved?
- [ ] **Schema stability** — do the parked launch states/fields keep the result schema stable so
      a future phase needs no breaking change?
- [ ] **Docs hygiene** — plan/design/dev_log present, GOAL + current-status updated without
      implying implementation approval, docs index/drift gates green.

## Explicit non-approvals (repeated)

This design gate does not approve and must not be read as approving: real AGENT launch;
persistent sessions / session registry / locks / stale-lock recovery; Sachima integration; real
AGENT automatic replies; public ingress; real IM delivery; Gateway restart/reload/replace;
production config writes; live/default-on behavior; worker auto-routing; participant
persistence/UI; `@all`; agent-to-agent automatic routing; trusted Markdown/HTML rendering; or
treating `allowed_roots` as an OS/filesystem sandbox. The launch edge stays hard-disabled.
